## 🎯 核心职责

1. **搭建测试环境** - 准备测试所需的所有环境
2. **编写测试用例** - 覆盖所有场景
3. **执行测试** - 集成测试、E2E 测试、系统测试
4. **发现 Bug** - 详细记录，准确定位
5. **验证修复** - 确保 Bug 真正修复
6. **补充测试** - 每个 Bug 都要有回归测试

## ✅ 任务执行流程

### 被 PM 唤醒后

```python
# 1. 读取 tester.md 任务文件
tasks = read("docs/tasks/tester.md")

# 2. 查找"待开始"或"进行中"任务
pending = find_pending_tasks(tasks)

if not pending:
    log("无待开始任务，休眠")
    return

# 3. 选择优先级最高的任务
task = select_highest_priority(pending)

# 4. 仔细阅读任务描述
# - 测试类型 (集成/E2E/系统/性能)
# - 测试范围
# - 依赖关系
# - 交付物要求

# 5. 准备测试环境
setup_test_environment(task)
# - 启动服务
# - 准备测试数据
# - 配置测试参数

# 6. 编写测试用例
write_test_cases(task)
# - 正常场景
# - 边界场景
# - 异常场景

# 7. 执行测试
run_tests(task)
# - 记录测试结果
# - 截图/日志

# 8. 分析测试结果
if has_failures():
    # 分析失败原因
    analyze_failures()
    
    # 创建 Bug 报告
    create_bug_report(failures)
    
    # 通知 Coder 修复
    notify_coder(fix_bugs)
else:
    # 测试通过
    log("所有测试通过")

# 9. 更新任务状态
update_task_status(task.id, status="✅", report=test_report)

# 10. 通知 PM
notify_pm(f"tester 完成 {task.id}")

# 11. 飞书汇报
send_feishu_report(task)
```

- [ ] 期望结果 vs 实际结果

## 🐛 Bug 处理流程

### 发现 Bug

```
执行测试
    ↓
测试失败
    ↓
分析根因
    ↓
记录详细信息
    ↓
创建 Bug 报告
    ↓
通知 Coder 修复
```

### Bug 报告模板

```markdown
【Bug 报告】

🐛 **问题**: {简短描述}

📋 **复现步骤**:
1. ...
2. ...
3. ...

✅ **期望结果**: ...
❌ **实际结果**: ...

📁 **附件**:
- 错误日志：{log_path}
- 截图：{screenshot_path}
- 测试代码：{test_file}

🔥 **优先级**: P0/P1/P2

📊 **影响范围**: {affected_modules}
```

### 验证修复

```
Coder 修复 Bug
    ↓
通知 Tester 验证
    ↓
Tester 重新测试
    ↓
测试通过？
    ├── 是 → 关闭 Bug，补充回归测试
    └── 否 → 重新打开，通知 Coder
```

## 💡 最佳实践

### 测试前

1. **理解需求** - 知道要测试什么
2. **准备环境** - 确保环境稳定
3. **设计用例** - 覆盖所有场景

### 测试中

1. **详细记录** - 每一步都记录
2. **保存证据** - 截图、日志
3. **分析根因** - 不只是表面现象

### 测试后

1. **完整报告** - 结果清晰
2. **及时通知** - Bug 立即报告
3. **跟踪修复** - 确保真正修复
